INSIGHT 1

성패는 화면이 아니라 금액 계산 검증

실제 주문/환불/재결제/정산 사례 5~10건 없이는 정산 로직과 "자동 롤백"의 정의 자체가 불가능합니다. 현재 엑셀 양식·프로세스 흐름도와 함께 착수 전 최우선 요청 대상입니다.

INSIGHT 2

플랫폼 API는 '확인 후 자동화'

판매 플랫폼 목록·API 제공 여부가 미공개입니다. 3~4주차 연동 PoC로 데이터 구조·로그인 가능성을 선검증하고, 그 전까지는 엑셀 업로드를 1차 기준선으로 견적·일정을 고정합니다.

INSIGHT 3

"자동 롤백"·승인선이 최대 복잡도

결재→정산 연쇄 반영이 이 ERP의 차별화 핵심이자 최대 복잡도 지점입니다. 매출을 덮어쓰지 않는 이벤트 원장 구조로 어떤 해석(소급 반영/원상 복원)이든 수용합니다.

01 · CORE

환불 승인 한 번에
7가지가 정확히 함께
움직이는 ERP

"환불이나 재결제 발생 시 매출 집계가 꼬인다" — 발주 동기 그 자체를 겨냥합니다. 기안 시점에는 어떤 재무 데이터도 바꾸지 않고, 최종 승인 순간에만 연쇄 1~6을 단일 트랜잭션으로 반영합니다.

  • 연쇄 1~6은 @Transactional 단일 커밋 — 부분 성공이 구조적으로 불가능
  • 반려 시 데이터 무변경 보장 (BS0006) · 동시 승인은 낙관적 잠금(version)으로 차단
  • 알림·이메일(연쇄 7)은 커밋 후 비동기 + 재시도 3회 — 알림 실패가 재무 반영을 되돌리지 않음
하나라도 실패하면 전체 롤백

연쇄 도중 오류가 나면 "데이터는 변경되지 않았습니다"로 응답합니다. 반쯤 반영된 매출·정산이 존재할 수 없는 구조 — 이것이 이 프로젝트에서 아키텍처를 고른 기준입니다.

모호한 요구사항 Top 3 — 해석 A/B에 따라 견적이 달라집니다

§7 #1 일정 ±3주

외부 판매 플랫폼 목록·API 제공 여부

해석 A

공식 API 제공 플랫폼 소수(1~3개) → API 자동 수집 가능

해석 B

API 미제공·판매자센터만 존재 → 엑셀 업로드 또는 Playwright 크롤링, 5개+면 공수 급증

우리의 대응: 3~4주차 PoC 선검증 · 그 전까지 엑셀 기준선 고정

§7 #2 정산 재개발 +2주

부분환불 시 정산 재계산 규칙

해석 A

매출만 환불액 차감 · 외주비는 작업 수행 여부로 별도 판단

해석 B

환불 비율에 비례해 외주비·수수료·손익까지 일괄 재계산

우리의 대응: 판정 함수 교체형 설계 — 실사례 5~10건으로 1주차 확정

§7 #4 데이터 모델 근본 구조

"자동 롤백"의 정확한 의미

해석 A

승인된 수정/환불을 주문·정산에 소급 반영(재계산)

해석 B

A + 이미 반영된 결재를 취소하고 이전 상태로 복원하는 역처리 포함 (+1~2주)

우리의 대응: 이벤트 원장 — A/B 모두 수용 (B는 반대 부호 이벤트)

누락 가능 요구사항 — 견적 전 확인하지 않으면 무상 범위 확장 압력으로 돌아옵니다

기존 엑셀 데이터 마이그레이션 범위 백업·장애 대응 (RTO/RPO) 프리랜서 개인정보 보호 검수 기준·하자보수 기간 동시 사용자 규모·성능 목표

ℹ 뱃지에 마우스를 올리면 이유가 표시됩니다. 이 항목들은 공고 원문에 없지만 구현 시 반드시 합의가 필요합니다 (→ 요구사항 검토서 §8, 발주자 질의서 Q10~Q16).

  • 최근 AI/LLM 기반 업무 자동화 통합

ERP는 단순 CRUD가 아니라 업무 흐름과 데이터 정합성이 핵심이라 보고, 주문·배정·결재·환불·정산·손익이 하나의 흐름으로 이어지도록 설계합니다.

ℹ 위 실적은 실제 수행 경험만 기재했습니다 (→ 사용자 마일스톤 문서 §16.2~§16.4).

06 · QUESTIONS 보너스 자료

발주자님께 꼭 여쭙고 싶은 10가지

전체 질의서 17건(🔴 필수 8 · 🟡 중요 8 · 🟢 선택 1) 중 핵심 10건 — "미확인/모름"도 유효한 답변이며, 그 경우 엑셀 업로드 기준선으로 견적을 고정합니다

Q1 🔴 필수 외부 판매 플랫폼의 목록·개수와 각 플랫폼의 공식 API 제공 여부는?

선택지: A) REST API + OAuth 등 공식 API 제공 / B) 판매자센터 로그인 + 엑셀 다운로드만 / C) 없음·미확인 — 플랫폼별 A/B/C 표기 요청

영향: 자동화/수동화 범위 전체와 주문 소스 ERD 결정. 무리한 전 플랫폼 자동화 시 일정 +3주 이상. 3~4주차 PoC 대상 선정에 직결.

Q2 🔴 필수 플랫폼 결제 내역(결제·수수료·환불·재결제)은 어떤 방식으로 확보 가능한가요?

선택지: A) 정산 API 제공 / B) 판매자센터 정산 엑셀 다운로드 / C) 미확인

영향: 정산 모듈 테이블 설계(수수료·환불 필드)의 전제. 확인 지연 시 11~12주차 정산 개발에서 ERD 재작업 위험 +2주.

Q3 🔴 필수 부분환불 발생 시 재계산 범위는 어디까지인가요?

선택지: A) 매출만 환불액 차감, 외주비는 작업 수행 여부로 별도 판단 / B) 환불 비율에 비례해 외주비·플랫폼 수수료·손익까지 일괄 재계산

영향: 정산 엔진 핵심 알고리즘. 착수 후 변경 시 정산 재개발 +2주. 실제 환불 사례 2~3건과 함께 답변 요청 (Q14 연계).

Q4 🔴 필수 "결재 승인 완료 시 자동 롤백 및 업데이트"는 어느 쪽 의미인가요?

선택지: A) 승인된 수정/환불 내용을 주문·매출·정산에 소급 반영(재계산) / B) A + 이미 반영된 결재를 취소하고 이전 상태로 복원하는 역처리 기능까지 포함

영향: 이벤트 원장 설계로 A/B 모두 수용 가능하나, B는 취소 결재 UI + 역방향 보상 이벤트가 추가 범위 +1~2주.

Q5 🔴 필수 결재 승인선 구조는 어떻게 되나요? (고정 vs 조건부)

선택지: A) 결재 유형별 고정 승인선 (예: 환불 = 팀장→재무→대표) / B) 금액 구간·부서·주문 유형 조건부 승인선 + 대결(위임)/전결 필요

영향: 결재 엔진 복잡도 2~3배 차이. 설정형 결재선으로 설계하되 B 전체를 1차 반영 시 +2주. 대결/전결은 필요 여부만 우선 확인.

Q6 🔴 필수 프리랜서/인플루언서가 시스템에 로그인하나요?

선택지: A) 아니오 — 내부 임직원 전용, 프리랜서는 관리 대상 데이터 / B) 예 — 작업 확인·지급 명세서 조회용 외부 포털 필요

영향: B면 사용자 수·권한 체계·화면·개인정보 접점이 대폭 증가 +3~4주 → 1차 범위 제외 후 Phase 2 분리 권고.

Q10 🔴 필수 전자세금계산서 발행 벤더와 인증 상태는 어떻게 되나요?

선택지: A) 이미 사용 중인 ASP 있음 (팝빌·바로빌 등 — 벤더명 회신 요청) / B) 미선정 — 홈택스 직접 발행 중

영향: B면 벤더 선정·계약·사업자 인증서 절차의 대기 시간 발생. 1차는 발행 요청/상태 관리까지, 완전 자동 발행은 2차 권고. 벤더 이용료 부담 주체도 확인.

Q14 🔴 필수 현재 엑셀 양식 전체 + 실제 주문/부분환불/재결제/정산 사례 5~10건을 계약 후 1주 내 제공 가능한가요?

선택지: A) 예 — 1주 내 제공 (개인정보 마스킹 가능) / B) 준비 필요 — 제공 가능 시점 회신 (N주)

영향: 정산 로직·"자동 롤백" 정의 검증의 유일한 근거 데이터. 지연되면 1개월 차 기획/디자인 완료 마일스톤이 그대로 밀립니다 (일정 1:1 지연).

Q11 🟡 중요 공식 API가 없는 플랫폼에 대해 판매자 계정 제공 + 브라우저 자동화(Playwright 크롤링)를 수용하시나요?

선택지: A) 예 — 계정 제공, 약관·차단 리스크 인지·수용 (실패 시 엑셀 대체 경로 상시 유지) / B) 아니오 — 해당 플랫폼은 엑셀 업로드로 운영

영향: 크롤링 워커(Node.js Playwright) 포함 여부. A여도 1차는 안정성 검증된 1~2개 플랫폼만 — 화면 변경·캡차 운영 리스크는 잔존.

Q16 🟡 중요 검수 완료 판정 기준과 하자보수 기간을 계약서에 명시하는 방식에 동의하시나요?

선택지: A) 예 — 15주차 통합 테스트 시나리오(주문→배정→결재→정산→대시보드) 통과를 검수 기준으로 하고 무상 하자보수 N개월 명시 / B) 자체 검수 기준 별도 보유 (공유 요청)

영향: 검수 기준 부재 시 오픈 지연·분쟁 리스크. 14주차 기능 개발 Freeze(신규 요구는 Phase 2 백로그) 원칙도 함께 합의.

ADR-001 채택

모듈형 모놀리식 (MSA·Kafka·과도한 DDD 배제)

7연쇄가 서비스 경계를 넘으면 분산 트랜잭션/사가 필요 — 모놀리식 + @Transactional이 원자성을 가장 값싸게 보증. 12개 모듈 패키지 경계는 유지해 향후 분리 여지 보존.

ADR-002 채택

환불/재결제는 덮어쓰기가 아니라 이벤트 기록 (매출 원장)

original_amount 불변 · 결제/환불/재결제를 부호 이벤트로 적재 · 매출은 항상 합산. "자동 롤백"이 소급 반영이든 원상 복원이든 둘 다 수용 — 어느 시점 매출도 재현 가능.

ADR-003 채택

재무 반영은 결재 최종 승인 후에만 — 7연쇄 단일 TX + 알림 분리

기안 시점 재무 무변경. 연쇄 1~6 단일 트랜잭션, 연쇄 7(알림)은 커밋 후 비동기 + 재시도 3회. 동시 승인은 낙관적 잠금으로 차단 — 부분 성공이 구조적으로 불가능.

ADR-004 채택

AI 파싱 결과는 후보 데이터로 저장, 담당자 검수 후 반영

Phase 2+ LLM 파싱은 ai_parse_results(후보)에만 적재 — 검수·승인 후 본 테이블 반영. LLM 오추출이 재무 데이터에 직행하지 않도록. 개인정보 마스킹 후 외부 전송.

ADR-005 채택

크롤링/RPA는 Node.js Playwright 워커로 분리

크롤링의 불안정성(화면 변경·캡차)을 프로세스 경계로 격리. 워커 장애가 ERP 가용성에 무영향 · 실패 시 엑셀 업로드 대체 경로 상시 유지 · 3~4주차 PoC 선검증.

ADR-006 채택

승인선은 설정 데이터로 관리 (하드코딩 금지)

approval_line_rules(결재 유형 × 금액 구간 → 승인 역할)를 설정 테이블로. 정책 변경 = 데이터 수정 — 승인선 미확정 리스크를 설계로 흡수. 대결/전결은 범위 제외 명시.

ADR-007 채택

단가 스냅샷 + 정산 확정 후 조정 이벤트

배정 시점 단가를 rate_snapshot으로 고정. 정산은 DRAFT→CONFIRMED→PAID 상태기계 — 확정 후 변경은 결재 승인 경유 조정 이벤트로만. 지급명세서 금액이 언제나 재현 가능.

ADR-008 채택
